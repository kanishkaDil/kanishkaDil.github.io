01 Pipeline Script
What is Pipeline Script?
A Pipeline Script is a Groovy-based pipeline definition typed directly into the Jenkins job configuration UI. The script lives inside Jenkins — no external files needed.
Jenkins UI → Pipeline → Script
pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        echo 'Building...'
        sh 'mvn clean package'
      }
    }
    stage('Test') {
      steps {
        sh 'mvn test'
      }
    }
    stage('Deploy') {
      steps {
        echo 'Deploying...'
      }
    }
  }
}
🖊️ Stored in Jenkins
The Groovy script is saved inside the Jenkins job config — not in your source code repository.
⚡ Instant to set up
Just paste or type the script in the job's Pipeline section. No repository access required.
🔧 Editable in the UI
Changes are made directly in the Jenkins interface without touching any source files.
02 Pipeline Script from SCM
What is Pipeline Script from SCM?
Pipeline Script from SCM stores your pipeline definition as a Jenkinsfile inside your source code repository (Git, SVN, etc.). Jenkins fetches and runs it automatically on every build.
Jenkinsfile  (root of your repo)
// Jenkinsfile lives in Git alongside your code
pipeline {
  agent any

  stages {
    stage('Checkout') {
      steps {
        checkout scm
      }
    }
    stage('Build') {
      steps {
        sh 'mvn clean package'
      }
    }
    stage('Deploy') {
      steps {
        echo 'Deploying to prod'
      }
    }
  }
}
📁 Stored in Git
The Jenkinsfile lives in your repository — version-tracked just like any other source file.
🔗 Jenkins points to repo
Configure Repository URL + branch in the job. Jenkins clones the repo and reads the Jenkinsfile.
🔄 Auto-synced
Every commit can trigger a build using the exact pipeline definition at that commit's revision.
03a Advantages of Pipeline Script
Why use Pipeline Script?
⚡
Quick to start — zero repo setup
Paste a Groovy script and your pipeline is live immediately. No repository, no branch config, no Jenkinsfile to commit.
🧪
Great for prototyping and experiments
Try new stages, test plugin steps, or debug a build sequence without making any git commits.
🔒
No SCM access required
Ideal when Jenkins cannot or should not connect to a source repository — air-gapped environments, quick demos, etc.
🖱️
Simple UI-driven workflow
Everything lives in one place — the Jenkins job config screen. Suitable for small teams or solo developers just learning CI/CD.
🛠️
No external dependency
The pipeline runs even if the SCM server is down, since the script is stored inside Jenkins itself.
03b Advantages of Pipeline Script from SCM
Why use Pipeline Script from SCM?
📜
Version-controlled pipeline
The Jenkinsfile history is tracked in Git. You can see who changed what, when, and why — full audit trail for free.
🔍
Code review for the pipeline itself
Pull requests, diff views, and team approval apply to pipeline changes just like application code. No silent config edits.
🤝
Team collaboration and consistency
Every developer sees the same Jenkinsfile in the repo. No surprises about what the pipeline does — it is right there in the code.
🌿
Branch-specific pipelines (Multibranch)
With Multibranch Pipeline jobs, each branch can have its own Jenkinsfile — feature branches run different stages than main.
♻️
Disaster recovery and portability
If Jenkins is wiped, just point a new job at the same repo — the pipeline definition is safely in Git, not lost with the server.

05 When to Use Which
Choose the Right Approach
🖊️ Use Pipeline Script when…
→You are learning Jenkins and just getting started
→You need a quick demo or one-off automation job
→You are prototyping a new pipeline idea before committing it
→The project has no source repository or SCM access
→It is a solo / internal script with no team changes expected
→You need the pipeline to run even when the SCM is unreachable
📁 Use Pipeline Script from SCM when…
→The project is team-based and multiple devs will change the pipeline
→You want code review (PR/MR) on pipeline changes
→You need branch-specific pipelines via Multibranch jobs
→You need an audit trail for who changed the pipeline and when
→You want disaster recovery — rebuild Jenkins anytime from Git
→This is a production CI/CD pipeline — always prefer SCM
💡 Best practice: Always use Pipeline Script from SCM for production workloads. Treat your Jenkinsfile as code — commit it, review it, and version it alongside your application.
